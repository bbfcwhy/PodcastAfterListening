# Feature Specification: 參考 ui_reference 之 UI 風格設計

**Feature Branch**: `002-ui-reference-style`
**Created**: 2026-01-29
**Status**: Draft
**Input**: User description: "我希望參考ui_reference這個資料夾裡面的code來設計現在這個專案的ui風格設計，包含配色、風格、介面呈現方式等，但實際功能還是以現在所規劃的功能為主。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 一致的視覺風格 (Priority: P1)

訪客進入網站時，看到的整體配色與風格與參考專案（ui_reference）一致：暖色系背景、主文字與次要文字層級分明、按鈕與重點區塊使用統一的強調色，邊框與陰影低調不搶戲。

**Why this priority**: 這是本功能的核心目標，讓現有專案在「長相」上對齊參考設計，不改變既有功能。

**Independent Test**: 可透過並排比對現站與參考設計的配色與元件風格，確認主色、背景、文字、按鈕等視覺一致。

**Acceptance Scenarios**:

1. **Given** 訪客開啟網站首頁, **When** 頁面載入完成, **Then** 全站背景為參考設計的暖色底色、卡片與輸入框為白底、主文字與次要文字顏色符合參考層級
2. **Given** 頁面上有主要操作按鈕, **When** 訪客檢視, **Then** 主按鈕使用參考設計的 CTA 色、次要按鈕為 CTA 描邊樣式
3. **Given** 頁面上有分隔線或邊框, **When** 訪客檢視, **Then** 邊框與分隔使用參考設計的淡色、低存在感樣式

---

### User Story 2 - 介面呈現方式對齊參考 (Priority: P2)

網站的版面結構與呈現方式對齊參考：導覽方式（如側邊欄或頂部導覽）、區塊圓角與留白、區塊標題搭配圖示或標籤的樣式、列表與卡片的排版方式，在維持現有功能的前提下盡量與參考一致。

**Why this priority**: 除了顏色外，版面與元件呈現方式會影響整體「感覺」，是風格一致的一環。

**Independent Test**: 可透過檢查首頁、節目列表、單集詳情等頁面的版面與元件樣式，確認與參考設計的呈現方式一致。

**Acceptance Scenarios**:

1. **Given** 訪客瀏覽任一頁面, **When** 檢視導覽與版面結構, **Then** 導覽位置與樣式（如側邊欄、頂部列）與參考設計一致或合理對應
2. **Given** 頁面上有內容區塊（如節目大綱、業配、留言區）, **When** 檢視區塊標題與內文排版, **Then** 圓角、留白、標題樣式（如圖示+文字）與參考設計一致
3. **Given** 頁面上有列表或卡片（如節目卡、單集卡）, **When** 檢視排版, **Then** 卡片圓角、陰影、hover 效果與參考設計一致

---

### User Story 3 - 互動與狀態樣式一致 (Priority: P3)

滑過、選中、焦點等互動狀態，以及提示、警告等狀態的視覺表現與參考設計一致：hover 使用低干擾的底色、選中狀態使用 CTA 相關色、警示或錯誤使用參考的狀態色且僅小面積使用。

**Why this priority**: 互動與狀態樣式影響使用時的體感，屬於風格的一部分，但不影響功能正確性。

**Independent Test**: 可透過實際操作（滑過按鈕、選中導覽、觸發錯誤或警告）確認顏色與樣式符合參考。

**Acceptance Scenarios**:

1. **Given** 訪客將滑鼠移過可點擊元素, **When** hover 時, **Then** 背景或邊框變化使用參考設計的 hover 色、不使用高飽和刺眼色
2. **Given** 導覽或篩選有「目前選中」狀態, **When** 訪客檢視, **Then** 選中狀態使用參考的 selected 樣式（如 CTA 相關淡底）
3. **Given** 頁面出現警告或錯誤提示, **When** 訪客檢視, **Then** 使用參考設計的狀態色（warn）、且僅在必要小範圍使用

---

### Edge Cases

- 若參考設計中某元件在本專案沒有對應功能（例如參考有 AI 實驗室、本專案沒有），則該元件不需實作，其餘共用風格仍須一致。
- 若本專案有參考中沒有的頁面或區塊（例如後台、審核介面），其風格仍須沿用同一套配色與元件規則，不另創一套。
- 響應式斷點與版面可依本專案需求微調，但同一斷點下的配色、圓角、字級層級等須與參考一致。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 全站背景、卡片與輸入框等容器顏色必須與參考設計的設計 token 對應一致（如 canvas、surface、surface-muted）。
- **FR-002**: 主文字、次要文字、弱化輔助字的顏色與層級必須與參考設計一致（text-primary、text-secondary、text-muted）。
- **FR-003**: 邊框與分隔線必須使用參考設計的 subtle/strong 規則，且陰影僅在規定範圍內使用（如 shadow-sm）。
- **FR-004**: 主要 CTA 按鈕與次要按鈕的顏色與樣式必須與參考一致（bg-cta、border-cta）；標籤、篩選等輔助互動使用 info 色，不用作主按鈕。
- **FR-005**: 警示、錯誤、到期等狀態必須使用參考的 warn 色，且僅在對應情境下小面積使用。
- **FR-006**: Hover 與選中狀態必須使用參考的 hover/selected 色，不得使用純黑或高飽和亮色。
- **FR-007**: 版面與元件呈現（導覽結構、區塊圓角、標題樣式、卡片排版）必須與參考設計對齊，且不改變現有規劃之功能與資料流。
- **FR-008**: 本專案既有之所有功能（節目列表、單集詳情、留言、搜尋、聯盟行銷展示、後台等）維持不變，僅調整視覺與版面以符合參考風格。

### Key Entities

- **設計 token**：參考設計中定義的語意化顏色與樣式名稱（如 canvas、surface、cta、text-primary），對應到實際色碼與使用規則。
- **參考來源**：專案內 ui_reference 資料夾中的設計說明（如 DESIGN_TOKENS.md）、樣式檔與元件範例，作為風格對齊的依據。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者開啟現站與參考設計並排比對時，能一致辨識出相同的主色、背景層級與按鈕風格。
- **SC-002**: 全站無硬編色碼（如 #xxx）直接寫在介面元件上，皆透過設計 token 或主題變數引用。
- **SC-003**: 本專案既有功能清單（依 001 規格）均可正常使用，且行為與規格一致，僅視覺與版面改變。
- **SC-004**: 新訪客在三分鐘內能感受到全站視覺風格統一，無明顯「某一頁畫風不同」的斷裂感。驗證方式：以 quickstart.md 視覺檢查清單完成為準。

## Assumptions

- ui_reference 資料夾會保留在專案內，且 DESIGN_TOKENS.md（或同等設計說明）為權威參考；若參考專案日後更新 token，本專案可選擇是否跟進，但本次規格以目前參考內容為準。
- 字型可與參考一致（如 Noto Sans TC），若本專案已有字型設定且不影響可讀性，可保留或與參考對齊，以不增加額外授權成本為原則。
- 圖示庫（如 Lucide）與參考相同或視覺上相容即可，不強制同一套圖示。
